## ko

Amazon S3는 **"인터넷의 파일 시스템"에 가장 근접한 객체 저장소**다. 2006년 AWS 최초 서비스로 출시된 뒤 지금까지 사실상 표준. 파일(객체)을 "버킷"에 넣으면, 11 9s(99.999999999%)의 내구성으로 보관되고 HTTP로 어디서든 접근 가능하다. 용량 상한 없음 — 단일 객체 5TB, 총량 무제한.

S3가 바꾼 사고방식은 "파일 스토리지 = 디스크"에서 "파일 스토리지 = API"로의 전환. 그래서 [[serverless]]·[[microservices]]·[[datalake]] 모두의 기반 블록이 된다. 정적 웹사이트 호스팅, 이미지·동영상 CDN 오리진, 백업 아카이브, 데이터 호수 저장, 로그 덤프, ML 데이터셋 — 용도가 끝없다.

### 스토리지 클래스 (비용/지연 트레이드오프)
- **Standard** — 자주 접근, 밀리초 지연, 최고가
- **Intelligent-Tiering** — AWS가 접근 빈도 보고 자동 분류 (default 추천)
- **Standard-IA / One Zone-IA** — 가끔 접근, 저장은 싸고 조회는 비쌈
- **Glacier / Deep Archive** — 수시간~12시간 조회, 가장 저렴. 장기 백업
- **Express One Zone** — 단일 AZ, 초저지연 (ML 학습 등)

### 언제 쓰나
- **정적 자산** — 이미지·비디오·JS·CSS. CloudFront와 붙이면 CDN
- **이벤트 주도** — 업로드 → [[lambda]] 트리거 → 썸네일·변환·[[kinesis]] 발송
- **[[datalake]] 기반** — Parquet·CSV·JSON 원본 저장, [[athena]]로 직접 질의
- **백업·스냅샷** — DB 덤프, [[iac]] state, 이미지 아카이브
- **로그 수집** — CloudWatch·ALB·CloudFront 로그가 S3로 흘러감

### 쉽게 빠지는 함정
- **퍼블릭 버킷 사고** — 실수로 Public 설정해 데이터 유출. Block Public Access를 *계정 레벨*에서 기본 ON
- **스토리지 클래스 방치** — 로그·백업을 Standard에 두면 장기적으로 가장 비싼 실수. Lifecycle policy로 자동 이관
- **수백만 작은 파일** — [[athena]] 쿼리가 메타데이터 부하로 느려짐. 파일 크기 최소 128MB 목표, 작으면 compaction
- **전송 비용 무시** — 같은 리전 EC2에서 다른 리전 S3로 전송은 초당 과금 없어 보이지만 월말에 폭탄. VPC 엔드포인트로 줄이기
- **LLM이 "버킷을 public으로 설정해" 제안** — [[hallucination]]의 보안 버전. [[review-mindset]]로 차단
- **버킷 네이밍 충돌** — S3 버킷 이름은 *전 세계 유일*. 조직 prefix 규칙 미리 정해야 함

### 연결
[[serverless]]·[[lambda]]·[[ecs]]의 저장 레이어. [[datalake]]의 토대이자 [[dw]] ETL의 중간 저장. [[cost]] 관점에서 "저장은 싸고 전송·조회가 비싸다"는 모델 이해 필수. [[iac]]와 짝 — S3 버킷은 CloudFormation/Terraform으로 만드는 게 기본. [[monitoring]]으로 요청 수·스토리지 크기·4xx/5xx 추적.

## en

Amazon S3 is **the object store closest to "a filesystem for the internet."** Launched in 2006 as AWS's first service, it's the de facto standard. Put a file (object) in a "bucket" and it persists with 11 9s (99.999999999%) durability, reachable over HTTP from anywhere. No overall capacity limit — 5 TB per object, unlimited total.

S3 shifted the mental model from "file storage = disk" to "file storage = API," which is why [[serverless]], [[microservices]], and [[datalake]] all use it as a foundation block. Static website hosting, image/video CDN origin, backup archive, data-lake storage, log dumps, ML datasets — the use cases never stop.

### Storage classes (cost/latency trade-offs)
- **Standard** — frequent access, millisecond latency, most expensive
- **Intelligent-Tiering** — AWS observes access patterns and auto-tiers (recommended default)
- **Standard-IA / One Zone-IA** — infrequent access, cheap to store but expensive to retrieve
- **Glacier / Deep Archive** — hours to half-day retrieval, cheapest. Long-term backup
- **Express One Zone** — single-AZ, ultra-low-latency (ML training etc.)

### When to use
- **Static assets** — images, video, JS, CSS. Fronted by CloudFront for a CDN
- **Event-driven** — upload → [[lambda]] trigger → thumbnail, transform, publish to [[kinesis]]
- **[[datalake]] foundation** — Parquet, CSV, JSON raw storage queried directly by [[athena]]
- **Backup and snapshot** — DB dumps, [[iac]] state, image archives
- **Log collection** — CloudWatch, ALB, CloudFront logs all flow into S3

### Common pitfalls
- **Public-bucket accidents** — a mis-set Public permission leaks data. Keep Block Public Access on at *account level* by default
- **Neglected storage class** — leaving logs and backups in Standard is the most expensive mistake long-term. Use lifecycle policies for auto-tiering
- **Millions of tiny files** — [[athena]] queries slow down under metadata load. Target at least 128 MB per file; compact otherwise
- **Ignoring transfer cost** — cross-region transfer from EC2 to S3 has no per-second fee but surprises at month end. Reduce via VPC endpoints
- **LLM suggests "make the bucket public"** — the security flavor of [[hallucination]]. Block via [[review-mindset]]
- **Bucket name collisions** — S3 bucket names are *globally unique*. Decide org-prefix conventions early

### How it connects
The storage layer under [[serverless]], [[lambda]], [[ecs]]. The foundation of [[datalake]] and the intermediate stage for [[dw]] ETL. From a [[cost]] view, understanding "cheap to store, expensive to transfer and query" is non-negotiable. Pairs with [[iac]] — buckets made via CloudFormation / Terraform by default. [[monitoring]] tracks request count, storage size, and 4xx/5xx.

## ja

Amazon S3は**「インターネットのファイルシステム」に最も近いオブジェクトストレージ**。2006年にAWSの最初のサービスとして出て以来、事実上の標準。ファイル(オブジェクト)を「バケット」に入れると11 9s(99.999999999%)の耐久性で保管され、HTTPでどこからでもアクセスできる。総容量の上限なし — 単一オブジェクト5TB、総量無制限。

S3が変えた発想は「ファイルストレージ=ディスク」から「ファイルストレージ=API」への転換。だから[[serverless]]・[[microservices]]・[[datalake]]のすべての基盤ブロックになる。静的Webサイトホスティング、画像・動画CDNオリジン、バックアップアーカイブ、データレイク保存、ログダンプ、MLデータセット — 用途は尽きない。

### ストレージクラス(コスト/レイテンシのトレードオフ)
- **Standard** — 頻繁アクセス、ミリ秒遅延、最高価
- **Intelligent-Tiering** — AWSがアクセス頻度を見て自動分類(既定推奨)
- **Standard-IA / One Zone-IA** — たまにアクセス、保存は安く取り出しは高い
- **Glacier / Deep Archive** — 数時間〜半日取り出し、最安。長期バックアップ
- **Express One Zone** — 単一AZ、超低遅延(ML訓練など)

### いつ使うか
- **静的アセット** — 画像・動画・JS・CSS。CloudFrontと組めばCDN
- **イベント駆動** — アップロード → [[lambda]]トリガ → サムネ・変換・[[kinesis]]送信
- **[[datalake]]の基盤** — Parquet・CSV・JSON原本を保存、[[athena]]で直接問い合わせ
- **バックアップ・スナップショット** — DBダンプ、[[iac]] state、イメージアーカイブ
- **ログ収集** — CloudWatch・ALB・CloudFrontログがS3に流れる

### はまりやすい罠
- **パブリックバケット事故** — 誤ってPublic設定してデータ漏洩。Block Public Accessを*アカウントレベル*で既定ON
- **ストレージクラスの放置** — ログ・バックアップをStandardに置きっぱなしにするのは長期的に最大の失敗。Lifecycle policyで自動移行
- **数百万の小ファイル** — [[athena]]クエリがメタデータ負荷で遅くなる。1ファイル最低128MBを目標、小さければcompaction
- **転送コスト無視** — 同リージョンEC2から別リージョンS3への転送は秒課金には見えないが月末に爆弾。VPCエンドポイントで削減
- **LLMが「バケットをpublicに」と提案** — [[hallucination]]のセキュリティ版。[[review-mindset]]で遮断
- **バケット名衝突** — S3バケット名は*全世界で一意*。組織プレフィックス規則を事前に決める

### 繋がり
[[serverless]]・[[lambda]]・[[ecs]]の保存層。[[datalake]]の土台であり[[dw]] ETLの中間保存。[[cost]]視点で「保存は安く、転送・照会が高い」モデルの理解が必須。[[iac]]と対 — S3バケットはCloudFormation/Terraformで作るのが基本。[[monitoring]]でリクエスト数・ストレージサイズ・4xx/5xxを追う。
